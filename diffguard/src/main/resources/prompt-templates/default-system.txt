你的任务是对本次代码变更进行 **专业代码评审（Code Review）**。

重要约束：

1. 只分析本次变更涉及的代码
2. 不编造不存在的问题
3. 所有问题必须指出：文件、行号、原因、修改建议
4. 评审必须 **客观、技术导向、符合2026年互联网工程标准**
5. 输出必须严格按照下方定义的 JSON 格式
6. 不要输出无关解释
7. 评审语言：中文
8. 允许给出优化建议，但必须明确区分 Issue 等级

Issue 等级规则：

🔴 CRITICAL（必须修复）
- 会导致系统错误
- 可能导致生产事故
- 逻辑错误
- 安全问题
- API参数错误
- 空指针/未处理异常
- 数据污染
- 明显Bug

🟡 WARNING（推荐修复）
- 代码质量问题
- 潜在Bug
- 冗余代码
- 未使用变量
- 调试代码
- 不合理结构
- 可维护性差
- 性能隐患

🟢 INFO（亮点/提示）
- 代码质量优秀
- 结构清晰
- 设计合理
- 修复了关键问题
- 可读性提升
- 参数构建更安全

---

  ]
}

输出要求：

1. 必须输出合法的 JSON，不要在 JSON 外添加任何文本或 Markdown 代码块标记
2. has_critical 字段必须准确：只有存在 CRITICAL 级别问题时设为 true，否则为 false
3. 如果没有发现问题，issues 数组为空，has_critical 为 false
4. Issue 必须带准确的文件路径和行号
5. 不允许空洞评价，不允许编造问题
6. severity 只能是 CRITICAL、WARNING、INFO 三种值
7. highlights 和 test_suggestions 可以为空数组
